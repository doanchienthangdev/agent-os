# Implementation Coordinator (CLA Phase 7)

## When to use

- After Phase 6 sprint-plan.md is approved AND `ops.capability_runs.state = 'implementing'`.
- Re-invoked by `/cla resume <id>` to pick up the next incomplete sprint.

## Inputs

- `capability_id`
- `sprint_plan_path` — `.archives/cla/<id>/sprint-plan.md`
- `spec_path` — `.archives/cla/<id>/spec.md`
- `draft_root` — `.archives/cla/<id>/draft/`
- `state_payload.completed_sprints` from `ops.capability_runs` — array of sprint IDs (e.g., `[1, 2]` means Sprint 3 is next)

## Process — multi-session, multi-week

### Step 1 — Read state, identify next sprint

Query `ops.capability_runs` for this capability. Extract:
- `state_payload.completed_sprints` (default `[]`)
- `state_payload.current_sprint` (default `1` if completed_sprints is empty)
- `state_since` — to compute idle time

If `(now - state_since) > interval '7 days'` AND `state_payload.current_sprint` not advanced, fire `failure_handling.phase_7_session_lost`: surface to founder ("Sprint N hasn't progressed in 7+ days — resume, abandon, or change plan?").

### Step 2 — Read sprint deliverables

From `.archives/cla/<id>/sprint-plan.md`, extract Sprint N's deliverables and acceptance criteria.

Cross-reference with `.archives/cla/<id>/draft/`:
- For each deliverable, find its draft (e.g., deliverable "Migration 00025 applied" → `.archives/cla/<id>/draft/migrations/00025_*.sql`).
- If a deliverable has no draft, surface to founder ("Sprint N deliverable X has no draft from Phase 5; create from scratch?").

### Step 3 — Delegate code work to `@cto`

For each deliverable, dispatch a focused Agent call to `@cto` (subagent):

> Implement deliverable: {name}.
> Source draft: `.archives/cla/<id>/draft/{relative-path}`.
> Target canonical location: {e.g., `supabase/migrations/00025_<name>.sql`}.
> Spec context: `.archives/cla/<id>/spec.md` § {section}.
> Constraints: must pass `pnpm check`; must follow repo conventions (file headers, idempotency, etc.).
> Open: a single change-set ready for git add (do NOT commit; orchestrator commits per sprint).

Sequential within a sprint to avoid file-conflict (each deliverable may touch overlapping files).

### Step 4 — Stage + commit + PR

Once all sprint deliverables are implemented in the canonical locations:
1. `git add` the changed files.
2. `git commit` with conventional message: `cla: <capability_id> sprint N — <name>`.
3. The husky pre-commit hook runs `pnpm check`. If it fails, the commit is blocked — fix the underlying issue (do NOT pass `--no-verify`).
4. `git push`.
5. `gh pr create --title "cla: <capability_id> sprint N (<name>)" --body "<auto-generated body referencing spec.md and sprint-plan.md>" --base main`.
6. Capture the PR URL.

### Step 5 — HITL Tier B per PR (handled by orchestrator)

`/cla` builds an `AskUserQuestion`:

```jsonc
{
  "question": "Sprint N PR ready: <url>. CI status: <pass|fail>. Cost actual vs estimated: <delta>. Approve to merge?",
  "header": "Sprint N PR",
  "multiSelect": false,
  "options": [
    { "label": "Merge", "description": "..." },
    { "label": "Request changes (specify)", "description": "..." },
    { "label": "Defer (specify why)", "description": "..." }
  ]
}
```

Founder merges via the GitHub UI or `gh pr merge`. The skill does not auto-merge.

### Step 6 — Persist sprint completion

After founder merges (orchestrator polls `gh pr view <url> --json state` until `MERGED`):
- UPDATE `ops.capability_runs` SET
  - `state_payload = jsonb_set(state_payload, '{completed_sprints}', state_payload->'completed_sprints' || to_jsonb(<N>))`
  - `state_payload = jsonb_set(state_payload, '{current_sprint}', to_jsonb(<N+1>))`
- INSERT `ops.events` (`${ORG_EVENT_NS}.capability.sprint_<N>_completed`).
- INSERT `ops.capability_phase_events`.
- INSERT `ops.cost_attributions` for the LLM/CTO cost of the sprint.
- Update `ops.capability_runs.actual_cost_setup_usd` (running total) and `actual_founder_hours` if founder reported hours.

### Step 7 — Loop or finalize

If more sprints remain (`current_sprint <= total_sprints`): return to Step 2 for next sprint. Multi-session — the founder may end the Claude Code session and resume later via `/cla resume <id>`.

If all sprints complete:
- Run final E2E test (per spec.md § 6 acceptance criteria).
- UPDATE `ops.capability_runs` SET `state = 'deployed'`, `state_since = now()`, `deployed_at = now()`, `current_phase = 8`.
- INSERT `ops.events` (`${ORG_EVENT_NS}.capability.deployed`).
- Hand off to `catalog-updater` (Phase 8).

## Multi-session resume semantics

`/cla resume <id>` calls this skill. The skill:
1. Reads `state_payload.current_sprint` from `ops.capability_runs`.
2. Re-runs Phase 0 drift check (state may have changed during the gap).
3. Validates the draft folder + working files are still consistent (e.g., `git diff` shows no surprise edits since last sprint).
4. Resumes at Step 2 for the current sprint.

If `main` has moved significantly since the architect's drafts were written (e.g., conflicting migrations in `supabase/migrations/`), surface to founder with "rebase drafts or re-run Phase 5?".

## Outputs

- N PRs in git history (one per sprint).
- Drafts under `.archives/cla/<id>/draft/` get COMMITTED to their canonical locations during this phase (not before).
- `ops.events` trail (one event per sprint completion + final `deployed`).
- Updated `ops.capability_runs.state_payload.completed_sprints[]`, `actual_cost_setup_usd`, `actual_founder_hours`.

## State transition

`implementing` (per sprint) → `deployed` (after final sprint + E2E).

## HITL

**Tier B per PR**. Founder merges; skill never auto-merges.

## Failure modes

| Symptom | Response |
|---|---|
| Sprint deliverable has no draft | Surface to founder; offer "create from scratch via @cto" or "skip + manually". |
| Husky pre-commit fails | Block commit; surface validator output to founder; iterate the deliverable until clean. |
| CI fails post-push | Same — fix and re-push; do NOT auto-merge with red CI. |
| Founder requests changes | Re-dispatch `@cto` with the changes baked in; new commit; new PR review round. |
| Idle > 7 days | Fire `failure_handling.phase_7_session_lost`; surface to founder. |
| `main` diverged significantly | Offer rebase OR Phase 5 re-architect. |
| Final E2E fails | Hold state at `implementing`; do NOT advance to `deployed`. |

## LLM mode awareness

- **Subscription / Hybrid / Full API:** Same flow; cost dominated by `@cto` per-deliverable calls.
- **Fallback (no API):** Skill cannot drive `@cto` invocations; founder implements manually using draft as starting point.

## Cost estimate

- Anthropic API: ~$0.50-2.00 per sprint depending on # of deliverables (each `@cto` call ~ $0.20-0.50).
- Founder time: per sprint estimate from sprint-plan.md (typically 4-8 hours per sprint).
- Cost-bucket: starts being attributed to `<capability_id>` (the per-capability bucket) for any post-deploy ongoing operation.

## Mode awareness (v1.1 — `cla-update-mechanism`)

| Mode | Skill behavior |
|---|---|
| `create` (default) | Full Process Steps 1-7 above. Multi-sprint, multi-PR via @cto. State persisted to `state_payload.completed_sprints[]`. |
| `fix` | **Single sprint, single PR.** Skip sprint loop logic. Delegate ONE deliverable to @cto. Open ONE PR. Founder Tier B per PR. NO multi-session expected (fix should fit in one session). |
| `extend` | Multi-sprint capable but typically 1 sprint. Same per-sprint loop as `create` but reads from `.archives/cla/<id>-extend-<session_id>/sprint-plan.md`. |
| `revise` | Same as `create` — multi-sprint, multi-week. Reads from `.archives/cla/<id>-revise-<session_id>/sprint-plan.md`. Multi-session resilient. |
| `tune` | **Not invoked.** Tune skips Phase 7 entirely; goes Phase 1 → Phase 8. catalog-updater handles the registry edit + single PR. |
| `deprecate` | **Not invoked.** Deprecate skips Phase 7; goes Phase 1 → Phase 3 → Phase 8. catalog-updater handles cleanup actions. |

**Lock awareness:** the `state_payload.session_id` set at Phase 0 (acquire) MUST be re-passed across sessions. On `/cla resume <id>`, this skill reads `state_payload.session_id` and uses it for any DB updates that touch lock-protected fields. The lock auto-expires after 24h, so resume after >24h gap requires re-acquisition (or `/cla force-unlock` if another session grabbed it).

**Final E2E gate** behavior unchanged across modes — must pass before state advances to `deployed`.
