# Contributing to agent-os

> agent-os is a substrate-only boilerplate. We accept improvements to the substrate (architecture, schemas, hooks, skills mechanism, consistency engine). We reject pillar-specific or org-specific content.

## What we accept

- **Substrate improvements** — bug fixes, performance improvements, schema corrections in `supabase/migrations/` (additive only — additions get a new migration number, never edit existing ones).
- **New generic hooks** — anything in `.claude/hooks/` that's truly cross-org useful (not org-specific).
- **New generic skills** — anything in `05-ai-ops/skills/` that mechanism-only and would help any org. Skills with org-specific assumptions belong in the user's fork.
- **Documentation improvements** — clearer explanations of the architecture, better examples in `00-charter/*.md` stubs, fixes to `governance/HITL.md` policy edge cases.
- **Init wizard improvements** — `scripts/init.cjs` enhancements, additional placeholder coverage, better idempotency.
- **DEPLOY.md improvements** — clearer guidance for the AI walking a founder through bootstrap.
- **JSON schema improvements** — stricter validation in `knowledge/schemas/`.
- **Test additions** — vitest tests in `tests/` that cover substrate behavior.

## What we reject

- **Org-specific pillar contents** — your filled-in `01-growth/sops/SOP-GROWTH-007.md` is not boilerplate material. Keep it in your fork.
- **Industry-specific role definitions** — the boilerplate has 3 generic roles. If you've added 8 roles for your org, that's specific to you.
- **Domain-specific knowledge files** — your customer personas, your competitor map, your product positioning are yours, not boilerplate.
- **Locale-specific compliance content** — the boilerplate's `00-charter/transparency.md` is a regulatory checklist scaffold. Filled-in obligations for your jurisdiction are yours.
- **Anything that introduces a `${PLACEHOLDER}` not documented in `notes/SUBSTITUTION-GUIDE.md`** — without the guide entry, the wizard can't substitute.

## How to contribute

1. Fork the repo.
2. Make your change in a feature branch.
3. Run `pnpm validate && pnpm test` locally.
4. Open a PR with:
   - **What:** one-line description
   - **Why:** the problem it solves (a concrete scenario where the substrate falls short)
   - **Risk:** what could go wrong; if it modifies an existing migration, hard reject (additive only)
   - **Substitution check:** if your change adds new templated content, you've added the variable to `notes/SUBSTITUTION-GUIDE.md`

PRs are reviewed by the source repo owner. Response SLA: best-effort; this is a side project.

## Coding conventions

- Match existing patterns. If you see `js-yaml` used elsewhere, don't introduce `yaml` package.
- Generated files (`*.generated.ts`) are regenerated by bundlers in `scripts/wave2-bundle-*.cjs`. Edit the YAML source, not the generated TS.
- Migrations: each new migration gets the next sequential number. Never edit a shipped migration; add a new one that ALTERs.
- Tier 1 files (`00-charter/`, `governance/`, `knowledge/manifest.yaml`) require schema validation via `pnpm validate`.

## Issue templates

See `.github/ISSUE_TEMPLATE/`. Use `bug.md` for substrate defects and `feature.md` for substrate enhancements.
